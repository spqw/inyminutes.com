Learn pnpm in Minutes

The fast, disk-efficient package manager for Node.js. Install packages, manage workspaces, and run scripts — with practical examples.

01 What is pnpm?

pnpm (performant npm) is a package manager for Node.js that stores packages in a global content-addressable store and hard-links them into your project. This means faster installs, less disk usage, and strict dependency isolation.

  • Fast — installs are 2-3x faster than npm due to hard linking and parallel resolution.
  • Disk efficient — packages are stored once globally, saving up to 70% disk space.
  • Strict — you can only import packages you explicitly declare (no phantom dependencies).
  • Monorepo-ready — built-in workspace support with powerful filtering.

02 Installation

# Option 1: Using corepack (recommended, ships with Node.js 16.13+)
corepack enable pnpm

# Option 2: Using npm
npm install -g pnpm

# Option 3: Standalone script (no Node.js required)
curl -fsSL https://get.pnpm.io/install.sh | sh -

# Verify installation
pnpm --version           # e.g. 10.5.0

# Pin pnpm version in your project (corepack)
corepack use pnpm@latest

Tip: With corepack, each project can specify its own pnpm version in package.json under "packageManager": "pnpm@10.5.0". Teammates automatically get the right version.

03 Package Management Basics

Installing dependencies

# Install all dependencies from package.json
pnpm install

# Add a production dependency
pnpm add react

# Add a dev dependency
pnpm add -D typescript @types/react

# Add an exact version (no ^ or ~)
pnpm add -E lodash@4.17.21

# Add an optional dependency
pnpm add -O fsevents

Removing and updating

# Remove a package
pnpm remove lodash

# Update a specific package
pnpm update react

# Update all packages (respecting semver ranges)
pnpm update

# Update all packages to latest (ignore semver ranges)
pnpm update --latest

# Check for outdated packages
pnpm outdated

Inspecting dependencies

# List installed packages
pnpm list

# List top-level only (no transitive)
pnpm list --depth 0

# Why is a package installed?
pnpm why react-dom

# Check for known vulnerabilities
pnpm audit

04 Running Scripts

# Run a script from package.json (no "run" needed)
pnpm build
pnpm test
pnpm dev

# Explicit "run" also works
pnpm run build

# Pass arguments to a script
pnpm test -- --watch

# Execute a package binary (like npx)
pnpm dlx create-next-app my-app
pnpm dlx degit user/repo my-project

# Run a local binary from node_modules/.bin
pnpm exec tsc --noEmit

Key difference: pnpm dlx = npx (download and execute). pnpm exec = run a binary already installed in node_modules.

05 Global Packages & Store

# Install a package globally
pnpm add -g typescript tsx serve

# List global packages
pnpm list -g

# Remove a global package
pnpm remove -g serve

# View the global store path
pnpm store path

# Clean up unreferenced packages from the store
pnpm store prune

# Check store integrity
pnpm store status

06 Workspaces (Monorepos)

pnpm has built-in monorepo support. Create a pnpm-workspace.yaml at the repo root:

# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'

Working with workspaces

# Install all dependencies for all workspace packages
pnpm install

# Add a dependency to a specific package
pnpm add react --filter @myorg/web

# Link a workspace package as a dependency
pnpm add @myorg/shared@workspace:* --filter @myorg/web

# Run a script in a specific package
pnpm --filter @myorg/web dev

# Run a script in all packages
pnpm -r build

# Run in all packages that match a pattern
pnpm --filter "./packages/**" test

# Run only in packages changed since last commit
pnpm --filter "...[HEAD~1]" build

# Add a dependency to the root workspace
pnpm add -w typescript

Tip: workspace:* always resolves to the local version. When publishing, pnpm replaces it with the actual version number automatically.

07 Lockfile & CI

# Install with frozen lockfile (CI mode — fails if lockfile is outdated)
pnpm install --frozen-lockfile

# Import lockfile from npm or yarn
pnpm import

# Generate lockfile without installing
pnpm install --lockfile-only

GitHub Actions example

# .github/workflows/ci.yml
steps:
  - uses: actions/checkout@v4
  - uses: pnpm/action-setup@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 22
      cache: 'pnpm'
  - run: pnpm install --frozen-lockfile
  - run: pnpm build
  - run: pnpm test

08 Advanced Features

Patching dependencies

# Create a patch for a package (opens editable copy)
pnpm patch express

# After editing, commit the patch
pnpm patch-commit ./path-to-modified-package

Overrides (force dependency versions)

// package.json — force all lodash to 4.17.21
{
  "pnpm": {
    "overrides": {
      "lodash": "4.17.21",
      "foo>bar": "2.0.0"
    }
  }
}

Configuration (.npmrc)

# .npmrc — project-level pnpm configuration

# Auto-install peers (default in pnpm 8+)
auto-install-peers=true

# Hoist specific packages (escape hatch for tools that need flat node_modules)
public-hoist-pattern[]=*eslint*
public-hoist-pattern[]=*prettier*

# Shamefully hoist everything (not recommended, but useful for migration)
shamefully-hoist=true

# Use a specific Node.js version via pnpm env
use-node-version=22.0.0

Manage Node.js versions

# Install and use a specific Node.js version
pnpm env use --global 22

# List installed Node.js versions
pnpm env list

# Remove a Node.js version
pnpm env remove --global 20

09 Migrating from npm / Yarn

# Step 1: Install pnpm
corepack enable pnpm

# Step 2: Remove old artifacts
rm -rf node_modules package-lock.json yarn.lock

# Step 3: Install with pnpm (generates pnpm-lock.yaml)
pnpm install

# Or import an existing lockfile first
pnpm import       # reads package-lock.json or yarn.lock
pnpm install

Expect: After switching, you may see errors for phantom dependencies — packages your code imports but doesn't declare in package.json. Fix these by adding them explicitly: pnpm add missing-package. This is pnpm catching real bugs.
